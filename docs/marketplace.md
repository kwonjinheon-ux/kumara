# Colink.nz Marketplace Design

Colink.nz 마켓플레이스는 뉴질랜드 한인 커뮤니티 안에서 개인 거래, 무료나눔, 업체판매, 공동구매를 빠르게 탐색하고 안전하게 거래하도록 설계한다.

## 핵심 방향

- Reddit형 좌측 메뉴로 큰 거래 흐름을 탐색한다.
- 데스크톱에서는 왼쪽 메뉴, 중앙 피드, 오른쪽 정보/광고 레일의 3열 구조를 사용한다.
- 모바일에서는 상단 카테고리 탭 대신 둥근 플로팅 햄버거 메뉴를 사용한다.
- 카테고리는 처음부터 과하게 노출하지 않고, 실제 기능이 준비된 항목만 보여준다.
- 추후 앱 확장을 위해 카테고리, 필터, 게시글 카드 정보는 API 기반 데이터 구조로 옮길 수 있게 유지한다.
- 화면은 빠른 탐색, 가격 가독성, 판매 상태 구분, 모바일 사용성을 우선한다.

## 데스크톱 3열 레이아웃

```text
[왼쪽 메뉴] [중앙 피드] [오른쪽 레일]
```

왼쪽 메뉴는 화면의 가장 왼쪽에 붙이고 카드처럼 떠 보이지 않게 한다. 중앙 피드와 좌우
영역 사이는 넓은 파란 여백을 두지 않고, Reddit처럼 얇은 구분선과 흰 배경으로 나눈다.

오른쪽 레일은 왼쪽 메뉴와 같은 폭을 기본으로 하며 다음 순서로 구성한다.

```text
상단: 고정 광고 영역
중단: 추천/인기 콘텐츠
하단: 자연스러운 광고 + 거래 안전 정보
```

오른쪽 상단 광고는 데스크톱에서 sticky로 유지한다. 추천 콘텐츠는 조회수와 댓글 수를
기준으로 간단히 정렬하고, 하단 정보는 거래 안전 안내나 파트너 소식처럼 피드를 방해하지
않는 내용만 배치한다.

중간 화면 폭에서는 오른쪽 레일을 숨기고 왼쪽 메뉴 + 중앙 피드 구조로 축소한다. 모바일에서는
왼쪽 메뉴를 플로팅 햄버거로 바꾸고, 오른쪽 레일 콘텐츠는 본문 아래의 카드형 정보 영역으로
내려 자연스럽게 이어지게 한다.

## 현재 최상위 메뉴

```text
마켓플레이스
├ 개인판매
├ 개인구매
├ 무료나눔
├ 업체판매
├ 공동구매
├ 인기글
├ 북마크
└ 내 판매글
```

준비되지 않은 카테고리는 화면에 노출하지 않는다. 공사중 항목은 내부 기획 문서에는 둘 수 있지만, 실제 사용자 화면에는 필요한 시점에만 추가한다.

## 모바일 메뉴

모바일에서는 좌측 메뉴를 화면에 계속 펼쳐두지 않는다. 대신 왼쪽에 둥근 플로팅
햄버거 버튼만 두고, 사용자가 누르면 데스크톱 메뉴와 같은 항목이 슬라이드/페이드
애니메이션으로 열린다. 버튼은 스크롤 위치와 무관하게 항상 접근 가능해야 한다.

```text
햄버거 버튼
→ 마켓플레이스 메뉴 패널
→ 개인판매 / 개인구매 / 무료나눔 / 업체판매 / 공동구매 / 인기글 / 북마크 / 내 판매글
```

가로 스크롤 카테고리 탭은 사용하지 않는다. 좁은 화면에서 메뉴가 피드를 밀어내거나
레이아웃을 흔들지 않도록, 메뉴는 오버레이 또는 독립 패널로 처리한다.

## 하위 카테고리

개인판매와 개인구매는 같은 하위 카테고리 구조를 사용한다.

```text
전체
가전제품
가구 / 인테리어
생활용품
주방용품
유아 / 아동용품
의류 / 패션
뷰티 / 건강
취미 / 스포츠
전자기기
책 / 문구
자동차용품
반려동물용품
티켓 / 상품권
기타
```

무료나눔, 업체판매, 공동구매는 추후 실제 게시글 수가 늘어난 뒤 별도 하위 카테고리를 세분화한다.

## 필터

상단 필터는 다음 항목을 기본으로 둔다.

```text
검색어
지역
물품 종류
판매 상태
정렬
거리
```

지역 필터:

```text
전체 / Auckland / Hamilton / Wellington / Christchurch / Tauranga / Dunedin / 기타
```

판매 상태:

```text
전체 / 판매중 / 거래중 / 판매완료
```

정렬:

```text
최신순 / 인기순 / 가격 낮은순 / 가격 높은순 / 댓글 많은순 / 조회수순
```

거리 필터:

```text
전체 / 5km 이내 / 10km 이내 / 25km 이내 / 50km 이내 / 100km 이내
```

거리 필터는 가입자의 프로필 지역과 세부 지역을 기준으로 한다. GPS 권한을 요구하지 않고,
현재 MVP에서는 지역명 중심 좌표를 사용해 게시글 거래 지역까지의 대략적인 직선 거리를
계산한다.

## 게시글 카드 필수 정보

```text
대표 썸네일
제목
댓글 수
작성자 닉네임
작성자 프로필
등록 시간
판매 상태
가격
지역
조회 수
```

댓글 수는 제목 옆에 한 번만 표시한다. 하단에 반복해서 `댓글` 텍스트를 노출하지 않는다.
댓글 수는 말풍선 아이콘이 아니라 작고 채워진 원형 숫자 배지로 표시한다.

가격은 카드에서 가장 잘 보여야 한다. 데스크톱에서는 오른쪽 컬럼에 가격을 크게 두고,
그 아래에 판매 상태를 배치한다. 모바일에서는 판매 상태를 썸네일 아래에 두고,
가격은 카드 오른쪽 하단에 강하게 정렬한다. 물품상태, 거래방식, 지역은 본문 정보
라인에 함께 표시한다.

모바일 카드는 Android Galaxy, iPhone, in-app browser에서 같은 정보 구조를 유지한다.
좁은 Galaxy viewport에서도 썸네일 / 본문 / 가격 영역의 3열 구조를 유지하고, 기기 폭이
작다고 가격 영역을 아래로 떨어뜨리거나 2열 카드로 바꾸지 않는다.

무료나눔과 가격제안은 금액 대신 텍스트로 표시한다.

카드 안에서는 조회수를 한 번만 보여준다. 작성자/시간 라인 옆에 조회수를 표시하고,
하단에 중복 조회수 줄을 다시 만들지 않는다.

끌어올린 글은 썸네일 왼쪽 위에 작은 불꽃 아이콘만 표시한다. 바깥 원형 배경,
붉은 그림자, `UP` 텍스트는 사용하지 않는다.

## 판매 상태

상태는 세 가지로 고정한다.

```text
판매중
거래중
판매완료
```

색상 기준:

```text
판매중: 초록
거래중: 주황
판매완료: 회색
```

작성자는 본인 게시글에서 상태를 빠르게 변경할 수 있어야 한다. 판매완료 이후에는 일반 수정은 잠그고, 관리자 권한으로만 예외 처리한다.

## 글쓰기

현재 구현 상세는 `docs/marketplace-write-mode-current.md`를 기준 문서로 둔다.

데스크톱에서는 상단 오른쪽에 `판매등록` 버튼을 둔다.

모바일에서는 마켓 검색창 오른쪽에 `판매등록` 버튼을 붙인다. 버튼과 검색창은
상단 네비게이션 검색창과 같은 높이를 유지하고, 버튼 글씨는 모바일에서 과하게
커지지 않도록 줄인다.

```text
마켓 검색 [          ] [판매등록]
```

글쓰기는 로그인한 사용자만 사용할 수 있다. 비로그인 사용자가 글쓰기를 누르면 로그인 페이지로 이동한다.

현재 글쓰기 폼 필드:

```text
물품 종류
제목
HTML 본문
가격 방식
가격
거래 지역
거래방식
물품상태
본문 이미지
대표 이미지 선택
연락 방식
등록 진행률
```

거래 유형은 글쓰기 화면에서 노출하지 않는다. 기본 글쓰기는 개인판매 기준으로 시작하고, 이후 메뉴/게시판 흐름에서 자동으로 결정한다.

거래방식:

```text
직거래
택배
직거래/택배
픽업만
협의
```

물품상태:

```text
새거
민트
상태 좋음
사용감 있음
완전중고
수리 필요
```

본문은 단순 textarea가 아니라 블로그 글쓰기처럼 HTML 형식으로 작성한다. 기본 편집기는 다음 기능을 제공한다.

```text
본문 / H2 / H3
굵게
기울임
밑줄
취소선
인용
목록
번호 목록
좌측 정렬
가운데 정렬
링크 / 링크 해제
구분선
서식 삭제
본문 중간 이미지 삽입
대표 이미지 지정
텍스트 색상 10가지
```

HTML 편집 도구는 처음부터 모두 펼치지 않는다. 모바일에서는 `HTML 편집툴 보기`
버튼을 눌렀을 때만 도구 모음을 펼쳐 직관성을 유지한다.

본문 편집기의 안내 문구는 사용자가 편집 영역을 누르면 사라져야 한다.

이미지는 두 방식으로 관리한다.

```text
1. 본문 중간 이미지
   - 블로그 글처럼 글 사이에 직접 삽입한다.
   - 선택한 이미지의 표시 크기를 작게 / 보통 / 크게 / 전체로 조절할 수 있다.
   - 이미지 파일 선택 후에도 마지막 커서 위치에 삽입한다.
   - 삽입된 이미지는 마우스로 직접 크기 조정할 수 있다.
   - 목록 썸네일에 사용할 경량 thumbnailDataUrl을 함께 만든다.

2. 본문 아래 첨부 이미지
   - 여러 장을 첨부할 수 있다.
   - 대표 이미지를 지정할 수 있다.
   - 대표 이미지는 이미지 위에 `대표` 배지로 표시한다.
   - 각 이미지는 오른쪽 위 `X` 버튼으로 개별 삭제할 수 있다.
   - 이미지 선택 버튼은 회색 계열의 미니멀 버튼으로 두고 hover 애니메이션을 적용한다.
   - 첨부 이미지 프리뷰는 마우스로 크기 조정할 수 있다.
```

현재 MVP에서는 업로드 전 클라이언트에서 이미지를 리사이즈한다. 본문/상세 표시는 최대
1280px JPEG, 목록 썸네일은 최대 420px JPEG로 생성한다. 운영 환경에서는 S3 업로드,
리사이징, MIME 검증, 악성 파일 검사를 연결한다.

대표 이미지는 게시글 카드 썸네일로 사용한다. 상세 페이지에서 데스크톱은 큰 대표
이미지와 아래 썸네일 목록을 사용하는 갤러리로 보여주며, 좌우 화살표는 이미지 크기와
무관하게 고정 위치에 둔다. 모바일에서는 복잡한 갤러리 대신 본문 흐름 안에 이미지를
세로로 자연스럽게 나열한다.

글쓰기 화면에서는 왼쪽 메뉴가 항상 따라오도록 sticky 구조를 사용한다. 데스크톱에서는 좌측 마켓 메뉴와 우측 작성 폼을 넓게 사용하고, 모바일에서는 왼쪽 메뉴를 햄버거 메뉴로 접어서 표시한다.

연락 방식은 Figma 스타일의 체크 옵션 패널로 표시한다. 흰색 옵션, 얇은 테두리,
8px radius, 민트 선택 상태, 작은 hover elevation을 사용한다. 카드형으로 과하게
부풀리지 않고, 체크 가능한 옵션 목록처럼 보이게 한다.

글 등록 중에는 화면 중앙에 modal progress overlay를 띄운다.

```text
글을 등록중입니다
이미지와 내용을 서버에 안전하게 저장하고 있습니다.
0% ~ 100%
```

등록 진행률은 `XMLHttpRequest.upload.onprogress`를 사용한다. 성공 시 상세 페이지로
이동하기 전까지 overlay를 유지하고, 실패 시 overlay를 닫고 오류 메시지를 보여준다.

## 글쓰기 성능 설계

현재 구현에서 적용된 성능 개선:

```text
- 이미지 원본을 그대로 보내지 않고 클라이언트에서 1280px/420px로 리사이즈
- 본문 이미지 원본을 gallery images payload에 중복 포함하지 않음
- 목록에서는 compact thumbnailDataUrl 우선 사용
- 목록/상세 SSR payload에서는 이미지 data URL을 제거하고 이미지 API URL로 대체
- 게시글/작성자/댓글 프로필/본문 inline 이미지는 /api/marketplace/posts/:postId/images/:imageId로 제공
- 이미지 응답은 public immutable cache header를 사용
- 게시글 JSON 읽기는 메모리 캐시와 파일 mtime으로 최적화
- 정렬된 게시글, postId map, public list item은 메모리 캐시로 재사용
- 상세 페이지가 새 postId를 못 찾으면 disk force refresh 후 404 판단
- 키워드 알림은 등록 응답 경로 밖에서 처리
```

10배 속도 개선을 위한 다음 설계:

```text
1. JSON 파일 저장소를 DB 테이블로 전환
2. base64 이미지 저장을 중단하고 object storage key만 저장
3. 이미지 파일은 presigned URL로 브라우저에서 S3/R2/GCS에 직접 병렬 업로드
4. 서버 POST는 제목/본문/가격/이미지 key만 받는 compact metadata 요청으로 축소
5. 썸네일 생성, 키워드 알림, 이메일은 job queue로 비동기 처리
6. post creation idempotency key를 추가해 재시도 중복 글 방지
7. 성공 응답은 post ID만 즉시 반환하고 상세 페이지는 DB에서 조회
```

목표:

```text
이미지 업로드 완료 후 글 등록 API P95 < 300ms
등록 직후 상세 페이지 transient 404 = 0
```

## 게시글 상세 읽기

사용자는 마켓플레이스 목록이나 마이페이지의 `내가 쓴 게시글`에서 제목을 눌러 게시글 상세 페이지로 이동할 수 있다.

```text
/marketplace/:postId
```

상세 페이지는 다음 정보를 보여준다.

```text
제목
판매 상태
작성자
등록 시간
조회수
가격
지역
거래방식
물품상태
대표 이미지
HTML 본문
본문/첨부 이미지 갤러리
```

본문은 저장된 HTML을 렌더링하되, 서버 저장 단계에서 script, style, 이벤트 핸들러, javascript: 링크를 제거해 기본적인 XSS 위험을 줄인다.

이미지 파일명이나 내부 파일 식별자는 사용자 화면에 노출하지 않는다.

작성자 정보 라인에는 둥근 프로필 이미지를 닉네임 앞에 표시하고, 닉네임, 작성일,
조회수를 작은 글씨로 정리한다.

가격, 지역, 거래방식, 물품상태는 파란 테두리 박스가 아니라 본문과 조화되는
미니멀 정보 영역으로 구성한다.

## 참여 기능

게시글 목록 카드에는 참여 정보를 반복해서 길게 보여주지 않는다. 목록에서는 제목 옆
댓글 수와 작성자 라인 옆 조회수만 유지한다.

게시글 상세 페이지에는 참여 기능을 모노톤 pill 버튼으로 제공한다.

```text
▲ 업
▼ 다운
♡/♥ 북마크
조회
댓글
```

북마크는 로그인한 사용자 기준으로 저장되며, 마이페이지의 `북마크` 탭에서 다시 읽을 수 있다.
북마크를 누르면 즉시 UI가 바뀌고, 빨간 하트가 퍼지는 짧은 애니메이션으로 피드백을 준다.

추천과 비추천은 사용자별로 하나만 유지한다. 같은 버튼을 다시 누르면 선택이 해제된다.
상세 페이지 참여 버튼은 기본 상태에서 과한 테두리를 보이지 않고, hover 시에만
현재 선택 가능한 버튼 형태가 드러나게 한다. 조회/댓글 메타 정보는 오른쪽 정렬을
기본으로 한다.

상세 페이지 거래 요약에는 로그인 사용자의 프로필 지역을 기준으로 판매 지역까지의
대략적인 거리를 표시한다. 위치 매핑이 불가능한 `기타` 지역이나 프로필 지역 미설정 상태에서는
거리 값을 숨긴다.

## 끌어올리기

본인 게시글의 수정, 삭제, 끌어올리기는 목록 카드에 노출하지 않는다. 상세 페이지나
내 글 관리 흐름에서만 제공한다. 끌어올리기 후 목록 썸네일에 불꽃 아이콘이 표시된다.

상세 페이지의 수정, 삭제, 끌어올리기 버튼은 지역 정보 박스와 같은 밝은 톤을 사용한다.
흰 배경, 연한 파란 테두리, 진한 텍스트로 처리해 판매 상태 pill보다 조용하게 보이도록 한다.

등급별 기본 주기:

```text
Iron: 7일
Silver: 6일
Gold: 5일
Platinum: 3일
Diamond: 2일
Master 이상: 1일
```

## 댓글

상세 페이지에는 댓글 영역을 둔다.

```text
댓글
└ 대댓글
   └ 대대댓글
```

댓글은 최대 3단계까지 허용한다. 기본 정렬은 추천 점수 기준이며, 같은 점수라면 최신 댓글을 위에 둔다.

댓글 UI는 Reddit형 threaded 구조를 따른다.

```text
작성자 프로필 + 닉네임 + 시간
댓글 본문
업 / 점수 / 다운 / Reply
└ 대댓글
   └ 대대댓글
```

Award, Share, 더보기 같은 현재 필요 없는 기능은 표시하지 않는다. 업/다운을 누르면
가운데 점수가 즉시 바뀌어야 한다.

댓글 입력은 큰 pill형 입력창 안에 등록 버튼을 포함한다. 입력 안내 문구는 빈 상태에서만
보이고, 사용자가 클릭하거나 입력하면 사라진다. 등록 버튼 hover에는 공통 버튼과 같은
사선 shine 애니메이션을 적용한다. 댓글이 없을 때의 안내 문구도 댓글 목록 바깥에
따로 떠 있지 않고 댓글 입력/댓글 영역 안에서 자연스럽게 처리한다.

## 성능 원칙

사용자가 북마크, 추천, 상태 변경, 댓글 등록 같은 버튼을 누르면 UI는 즉시 반응해야 한다.
서버 저장을 기다리느라 화면 반응이 늦어지지 않도록 optimistic update를 기본으로 한다.
MVP 로컬 JSON 저장 단계에서도 목록 조회는 가벼운 요약 데이터로 처리하고, 상세 페이지에서
필요한 전체 데이터를 가져온다.

## 운영 원칙

- 준비되지 않은 기능은 사용자 화면에 노출하지 않는다.
- 카테고리는 실제 게시글 수와 운영 필요성이 생길 때 추가한다.
- 개인 거래 안전 안내, 신고, 차단, 관리자 검토 흐름을 반드시 포함한다.
- 모바일에서 카드 정보가 복잡해지지 않도록 썸네일, 제목, 가격, 상태, 지역, 반응 수만 우선 보여준다.
